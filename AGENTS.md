# AGENTS.md

## プロジェクト概要

このリポジトリでは、React Native / Expo / TypeScript を使って、サウナ記録アプリ「サウナ行った」のMVPを開発する。

コンセプトは「サウナ記録をがんばらない」。
サウナ後でも迷わず使えるように、手動入力を最小限にし、選択式の入力を中心にする。

## 作業前に読む資料

実装や仕様変更の前に、次の順番で資料を確認すること。

1. `README.md`
2. `docs/MVP_specification.md`
3. `docs/Codex実装指示書.md`
4. `docs/実装タスク一覧.md`

`CODEX_PROMPT.md` と `docs/Codex初回プロンプト.md` は同じ初回指示を記載している。
両ファイル内の `docs/MVP仕様書.md` という表記は、実在する
`docs/MVP_specification.md` を指すものとして扱う。

仕様が食い違う場合は、具体的な画面・データ・操作仕様が記載されている
`docs/MVP_specification.md` を優先し、独断で機能を拡張しない。

## MVPの実装範囲

- サウナ記録の追加
- 記録一覧
- 記録詳細
- 記録編集
- 記録削除
- ローカル保存
- X投稿文の生成
- React Nativeの共有機能

入力項目は以下に限定する。

- 日付
- 施設名
- ととのい度
- セット数
- サウナの種類
- ひとことメモ

## MVPで実装しないもの

- ログイン、ユーザー登録、認証
- クラウド同期、バックアップ
- 施設検索、施設マスタ
- X APIとの直接連携
- 画像投稿
- カレンダー、ランキング、集計
- お気に入り、ダークモード
- その他、仕様書にない追加機能

将来機能を先回りして実装しない。必要になった時点で仕様を更新してから着手する。

## 技術方針

- React Native
- Expo
- TypeScript
- AsyncStorage
- 既存構成に応じて Expo Router または React Navigation

プロジェクト作成後は、実際の `package.json` と既存構成を優先する。
新しい依存関係は、MVP実装に必要なものだけ追加する。

以下の責務は分離する。

- 型定義
- 選択肢定数
- ローカル保存処理
- X投稿文生成処理
- 画面コンポーネント

推奨配置は以下とするが、Expo Routerなど既存テンプレートの構成に合わせて調整してよい。

```text
src/
├── components/
├── constants/saunaOptions.ts
├── features/saunaLogs/
├── screens/
├── storage/saunaLogStorage.ts
├── types/saunaLog.ts
└── utils/generateXPostText.ts
```

## 実装ルール

- 一度に大きく実装せず、動作確認できる小さな単位で進める。
- 型定義と選択肢定数を先に整える。
- 仕様書の値、ラベル、文言を独断で変更しない。
- 保存データは新しい記録が先に並ぶようにする。
- 施設名とととのい度を必須とし、その他は任意にする。
- 日付の初期値は端末上の今日の日付にする。
- 編集時は既存値を初期表示し、保存時に `updatedAt` を更新する。
- 削除前に確認ダイアログを表示する。
- X投稿文にメモを含めない。
- X APIは使わず、React Nativeの `Share` APIを使う。
- UIは余白とタップ領域を確保し、情報過多を避ける。
- メモ未入力など、任意項目を空のままでも自然に完了できる設計にする。

## 実装順

原則として以下の順番で進める。

1. 現在のプロジェクト構成と依存関係を確認する
2. 型定義を作成する
3. 選択肢定数を作成する
4. ローカル保存処理を作成する
5. 画面遷移と各画面を作成する
6. X投稿文生成処理を作成する
7. 共有機能を実装する
8. 型チェック、テスト、起動確認を行う

詳細な進捗は `docs/実装タスク一覧.md` に沿って管理する。

## 品質確認

変更後は、リポジトリに定義されたスクリプトを使い、可能な範囲で以下を確認する。

- TypeScriptの型チェックが通る
- lintが通る
- 関連テストが通る
- Expoアプリが起動時にクラッシュしない
- 記録の追加、再読込、一覧、詳細、編集、削除が動作する
- 必須項目のエラーが表示される
- 共有画面に仕様どおりの投稿文が渡される

実行できなかった確認項目や環境依存の確認は、作業報告に明記する。

## 変更時の注意

- ユーザーが作成した未関連の変更を削除、上書き、巻き戻ししない。
- 依頼と無関係なリファクタリングや依存関係更新を混ぜない。
- 仕様変更が必要な場合は、実装と対応するドキュメントを同じ変更内で更新する。
- 実装後は、変更ファイル、実装内容、確認コマンド、未確認事項を簡潔に報告する。
